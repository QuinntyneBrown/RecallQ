# L1 - High-Level Requirements

Project: **RecallQ** — a mobile-first web application for semantic (vector) search across contacts and the interactions a user has had with them.

Platform: **.NET** backend, **Angular** frontend. Design source of truth: `docs/ui-design.pen`.

All requirements use "shall" or "must" — never "should" or "may".

---

## L1-001: User authentication and account management
The system shall authenticate users and scope all contact, interaction, and embedding data to the authenticated account. No user shall be able to read or modify another user's data.

## L1-002: Contact management
The system shall allow an authenticated user to create, read, update, and delete contacts. A contact shall capture person information including at minimum: display name, initials, role/title, organization, tags, location, and an optional avatar.

## L1-003: Interaction tracking
The system shall allow an authenticated user to log interactions against a contact. Each interaction shall have a type (email, call, meeting, note), a timestamp, and free-text content. Interactions shall be searchable and displayed on a contact's activity timeline.

## L1-004: Vector semantic search
The system shall provide semantic (vector-similarity) search over both contact profile text and interaction text. A natural-language query shall return a ranked list of matching contacts with a similarity score between 0.00 and 1.00 and a short "why this match" snippet derived from the matching source text.

## L1-005: Conversational AI assistant ("Ask RecallQ")
The system shall provide an "Ask" mode where the user enters free-form natural-language questions about their network. The assistant shall answer using retrieval over the user's contacts and interactions and shall cite the specific contacts it grounded its answer on.

## L1-006: Smart Stacks (curated segments)
The system shall surface curated groupings of contacts ("Smart Stacks") on the home screen — for example "AI founders", "Intros owed", "Close friends" — each showing a count and opening a filtered list on tap.

## L1-007: Proactive AI suggestions
The system shall surface proactive AI suggestions on the home screen derived from recent interaction patterns (e.g., "You met 3 AI founders last week — shall I find similar investors?"). Each suggestion shall have a primary action and a dismiss action.

## L1-008: AI-generated relationship summaries
The system shall generate and display a per-contact relationship summary that synthesizes interaction history, sentiment, interaction count, and time since last contact. Summaries shall be refreshable on demand.

## L1-009: Contact detail view with activity timeline
The system shall provide a contact detail view presenting the contact's identity (name, role, org, tags), an AI relationship summary, a reverse-chronological activity timeline, and quick actions.

## L1-010: Quick actions from contact detail
The system shall expose four quick actions on the contact detail view: Message, Call, Intro, and Ask AI. Each action shall perform its operation or open its target without a full page reload.

## L1-011: Mobile-first responsive UI
The system shall be mobile-first. The UI shall render correctly and remain fully usable at the five viewport breakpoints XS (<576px), SM (≥576px), MD (≥768px), LG (≥992px), and XL (≥1200px). The default (unadorned) experience shall match the mobile design at 390px width.

## L1-012: Visual design system compliance with ui-design.pen
The implemented UI shall match the design system defined in `docs/ui-design.pen`: color tokens, typography (Geist / Inter / Geist Mono), spacing, corner radii, gradients, shadows, and the reusable components (Avatar 40/48/64, Button Primary, Button Ghost, Contact Card, Tag Chip, Tag Solid, Score High/Mid/Low, Search Bar, Ix Email/Call/Meeting/Note, Bottom Nav).

## L1-013: Security, privacy, and data protection
The system shall enforce authentication on every data endpoint, validate all inputs, protect against the OWASP Top 10, encrypt contact and interaction data at rest and in transit, and ensure embeddings and raw source text of one user are never exposed to another user or to any background job scoped to a different user.

## L1-014: Performance and scalability
The system shall return interactive vector search results within 400ms (p95) at up to 100,000 contacts and 1,000,000 interactions per user. Initial mobile page load shall be under 2.5s on a throttled 3G connection. The AI Ask answer shall begin streaming within 1.5s (p95).

## L1-015: Accessibility
The system shall meet WCAG 2.2 AA. All interactive elements shall be keyboard-reachable, have visible focus, and expose accessible names. Color contrast of text against its background shall meet AA minimums even within gradient-filled surfaces.

## L1-016: Observability, logging, and operations
The system shall emit structured logs, request/operation traces, and metrics for API latency, embedding-generation latency, vector-search latency, and LLM token usage. Errors shall be captured with correlation IDs. No personally identifiable contact data shall appear in logs.

## L1-017: Radically simple architecture
The implementation shall be radically simple. Backend: a single ASP.NET Core Web API project, EF Core against PostgreSQL with the `pgvector` extension, minimal API endpoints (or thin controllers), and no microservices, no message bus, no CQRS, no hand-rolled DI abstractions. Frontend: Angular with standalone components, Signals for state, the built-in HttpClient, and no NgRx or other global store. No code generator, no DDD layering, no repository/unit-of-work abstractions over EF Core.

## L1-018: Data import and embedding pipeline
The system shall support importing contacts and interactions (at minimum a manual "Add contact" and "Add interaction" flow, plus a CSV import for initial seeding). Embeddings shall be generated for new and updated records and stored in the vector column. Embedding regeneration shall be idempotent and resumable.
