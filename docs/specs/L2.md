# L2 - Detailed Requirements

Each L2 traces to exactly one L1. Acceptance criteria use Given/When/Then. Viewport breakpoints: **XS** <576px, **SM** ≥576px, **MD** ≥768px, **LG** ≥992px, **XL** ≥1200px.

Reference design: `docs/ui-design.pen`. Design-system tokens referenced with the exact names in the `.pen` file (e.g., `$accent-primary`, `$surface-secondary`, `$foreground-muted`).

---

## Authentication and accounts

### L2-001: User registration with email and password
**Traces to:** L1-001

The system shall allow a new user to register with an email address and a password. The email shall be unique per account. Passwords shall be hashed with a modern KDF (Argon2id or bcrypt with cost ≥ 12). The system shall never return the password hash.

**Acceptance Criteria:**
1. Given no account exists for `alice@example.com`, when the user submits a registration with that email and a valid password, then a new account is created and the response status is `201 Created`.
2. Given an account already exists for `alice@example.com`, when a registration is attempted with the same email, then the response is `409 Conflict` with no new account created.
3. Given a password shorter than 12 characters or missing a mix of letters and digits, when a registration is attempted, then the response is `400 Bad Request` with a validation error and no account is created.
4. Given any successful or failed registration, when the response is inspected, then no password or password hash appears in the response body or logs.

---

### L2-002: User login with bearer token
**Traces to:** L1-001

The system shall allow a registered user to log in and receive an access token (JWT, HttpOnly secure cookie, or opaque bearer). Invalid credentials shall produce a generic error and shall not reveal whether the email exists.

**Acceptance Criteria:**
1. Given a registered user with valid credentials, when they POST to the login endpoint with correct email and password, then the response is `200 OK` and includes an access token.
2. Given an email that exists but a wrong password, when login is attempted, then the response is `401 Unauthorized` with a generic "invalid credentials" message.
3. Given an email that does not exist, when login is attempted, then the response is `401 Unauthorized` with the same generic message as criterion 2 (no account-existence leak).
4. Given 5 failed login attempts within 60 seconds from the same IP for the same email, when a 6th attempt is made, then the response is `429 Too Many Requests`.

---

### L2-003: Authenticated data endpoints
**Traces to:** L1-001

Every endpoint returning contact, interaction, embedding, summary, or search data shall require a valid authenticated principal. Unauthenticated requests shall return `401` with no data leakage.

**Acceptance Criteria:**
1. Given no access token, when any `/api/contacts`, `/api/interactions`, `/api/search`, `/api/ask`, `/api/stacks`, or `/api/suggestions` endpoint is called, then the response is `401 Unauthorized` and the body does not contain any contact data.
2. Given an expired access token, when any data endpoint is called, then the response is `401 Unauthorized`.
3. Given a valid access token for user A, when a GET is made for a contact owned by user B, then the response is `404 Not Found` (not `403`, to avoid resource enumeration).

---

### L2-004: Logout invalidates session
**Traces to:** L1-001

The system shall provide a logout endpoint that invalidates the caller's session/token. Subsequent calls with the logged-out token shall be rejected.

**Acceptance Criteria:**
1. Given an authenticated user, when they POST to `/api/auth/logout`, then the response is `204 No Content` and the token is added to a server-side revocation state (or the HttpOnly cookie is cleared).
2. Given a token that has been logged out, when it is used on any protected endpoint, then the response is `401 Unauthorized`.

## Contact management

### L2-005: Create contact
**Traces to:** L1-002

The system shall allow an authenticated user to create a contact with: `displayName` (required, 1–120 chars), `initials` (required, 1–3 chars), `role`, `organization`, `location`, `tags` (array of strings), `emails` (array), `phones` (array), `avatarColorA`/`avatarColorB` (used to render the gradient avatar when no image is supplied).

**Acceptance Criteria:**
1. Given an authenticated user, when they POST a valid contact payload to `/api/contacts`, then a contact is created scoped to that user, the response is `201 Created` with the created contact, and an embedding job is enqueued for that contact.
2. Given a payload missing `displayName`, when POSTed, then the response is `400 Bad Request` with a validation error naming the missing field.
3. Given `displayName` longer than 120 characters, when POSTed, then the response is `400 Bad Request`.
4. Given a successful create, when the contact detail screen is rendered, then the hero shows the `displayName`, `role · organization`, and tags as chips matching the `heroTags` element of screen `3. Contact Detail` in `ui-design.pen`.

---

### L2-006: Read contact scoped to owner
**Traces to:** L1-002

The system shall return a contact only to its owning user.

**Acceptance Criteria:**
1. Given user A owns contact C, when user A GETs `/api/contacts/{C.id}`, then the response is `200 OK` with the contact body.
2. Given user A owns contact C and user B is authenticated, when user B GETs `/api/contacts/{C.id}`, then the response is `404 Not Found`.
3. Given a non-existent contact id, when GETed, then the response is `404 Not Found` and the body contains no internal identifiers or stack traces.

---

### L2-007: Update contact
**Traces to:** L1-002

The system shall allow the owner to update any editable field of a contact. Any change to text fields (`displayName`, `role`, `organization`, `tags`, `location`) shall trigger re-embedding of that contact.

**Acceptance Criteria:**
1. Given an authenticated owner, when they PATCH a contact with a new `role`, then the stored contact reflects the new role and an embedding-regeneration job is enqueued for that contact.
2. Given a PATCH that attempts to change `id` or `ownerUserId`, when the request is processed, then those fields are ignored and the response still reflects the original values.
3. Given a non-owner authenticated user, when they PATCH a contact they do not own, then the response is `404 Not Found`.

---

### L2-008: Delete contact cascades
**Traces to:** L1-002

Deleting a contact shall delete its interactions, its embeddings, and its AI relationship summary. Delete shall be irreversible (no soft-delete tombstones in v1) and shall be owner-scoped.

**Acceptance Criteria:**
1. Given an owner and a contact C with N interactions and 1 relationship summary, when the owner DELETEs `/api/contacts/{C.id}`, then the response is `204 No Content` and all interactions, embeddings, and the summary for C are removed from the database.
2. Given a non-owner, when they DELETE a contact they do not own, then the response is `404 Not Found` and no data changes.
3. Given a deleted contact id, when the owner GETs it, then the response is `404 Not Found`.

---

### L2-009: List contacts with pagination
**Traces to:** L1-002

The system shall return the authenticated user's contacts in a paginated list sorted by a specified field (default: most-recently-interacted).

**Acceptance Criteria:**
1. Given 150 contacts for user A, when user A GETs `/api/contacts?page=1&pageSize=50`, then the response contains 50 contacts, a `totalCount` of 150, and a `nextPage` of 2.
2. Given `sort=name`, when the list is returned, then contacts are ordered alphabetically by `displayName` (case-insensitive).
3. Given no contacts exist, when the list is requested, then the response is `200 OK` with an empty `items` array and `totalCount` of 0.

## Interaction tracking

### L2-010: Log interaction against a contact
**Traces to:** L1-003

The system shall allow an owner to log an interaction against a contact with `type` ∈ {`email`, `call`, `meeting`, `note`}, `occurredAt` (timestamp, required), `content` (free text, 0–8000 chars), and optional `subject`.

**Acceptance Criteria:**
1. Given owner A and a contact C, when A POSTs to `/api/contacts/{C.id}/interactions` with `type=call`, a valid ISO-8601 `occurredAt`, and content, then the response is `201 Created`, the interaction is persisted, an embedding job is enqueued for the interaction, and a relationship-summary-refresh job is enqueued for C.
2. Given `type` not in the allowed set, when POSTed, then the response is `400 Bad Request`.
3. Given content longer than 8000 characters, when POSTed, then the response is `400 Bad Request`.
4. Given a logged interaction, when the contact detail screen renders, then the interaction appears in the activity timeline styled with the matching design-system component (`Ix Email`/`Ix Call`/`Ix Meeting`/`Ix Note`) from `ui-design.pen`.

---

### L2-011: Timeline displays reverse-chronological interactions
**Traces to:** L1-003

**Acceptance Criteria:**
1. Given a contact with 24 interactions, when the contact detail screen is rendered, then the section header reads `Recent activity` and the "See all" affordance reads `See all 24` (matching the `tlHead` / `SudXW` text `See all 24` in `ui-design.pen`).
2. Given interactions, when the timeline renders, then items are ordered newest-first by `occurredAt`.
3. Given more than 3 interactions, when the contact detail screen is on XS viewport, then only the 3 most-recent interactions are shown inline and the remainder are reached via "See all".

---

### L2-012: Interaction pills on search results and Ask answers
**Traces to:** L1-003

Result cards and Ask-mode mini-cards shall display compact interaction pills (e.g., `Call · 2d`, `12 emails`, `Dinner · last wk`) matching the pill components in `ui-design.pen` (`Ix Email`, `Ix Call`, `Ix Meeting`, `Ix Note`).

**Acceptance Criteria:**
1. Given a contact with 3 interactions of different types, when its result card renders, then up to 3 pills appear, one per interaction type, each using the corresponding component.
2. Given a pill for a recent interaction, when the pill text is generated, then the relative time is rendered in short form (e.g., `2d`, `4d`, `last wk`, `Mon`).
3. Given a contact with multiple interactions of the same type, when the pill is rendered, then it shows an aggregate count (e.g., `12 emails`).

---

### L2-013: Update and delete interactions
**Traces to:** L1-003

**Acceptance Criteria:**
1. Given an owner, when they PATCH an interaction's content, then the stored content updates and an embedding job is enqueued for that interaction.
2. Given an owner, when they DELETE an interaction, then the response is `204 No Content`, the interaction and its embedding are removed, and a relationship-summary-refresh for the contact is enqueued.
3. Given a non-owner, when they attempt update or delete, then the response is `404 Not Found`.

## Vector semantic search

### L2-014: Natural-language search returns ranked contacts
**Traces to:** L1-004

The system shall accept a natural-language query string and return a ranked list of contacts whose profile or interaction embeddings are most similar to the query embedding. Similarity shall be cosine similarity in [0, 1].

**Acceptance Criteria:**
1. Given a user with indexed contacts and a query `"investors who liked AI tools"`, when POSTed to `/api/search` with `{ "q": "investors who liked AI tools" }`, then the response is `200 OK` with an ordered array of results, each containing `contactId`, `similarity` (0.00–1.00, two decimals), `matchedText`, and `matchedSource` ∈ {`contact`, `interaction`}.
2. Given the same query, when the response is rendered on screen `2. Search Results`, then the query chip displays `"investors who liked AI tools"` exactly (matching `LeAmW` in `ui-design.pen`).
3. Given no contacts or interactions exist, when the query is submitted, then the response is `200 OK` with an empty `results` array and a human-readable "no matches" state is rendered on the screen.

---

### L2-015: Search spans contact and interaction text
**Traces to:** L1-004

A single query shall evaluate similarity against both contact profile embeddings and interaction content embeddings, and shall return the best-matching source per contact (collapsing multiple interaction hits per contact to the single best one).

**Acceptance Criteria:**
1. Given a contact C whose profile has similarity 0.62 to the query and whose interaction I1 has similarity 0.91 to the query, when the query is executed, then the result for C reports `similarity=0.91` and `matchedSource=interaction`.
2. Given three interactions for contact C all above the threshold, when the response is built, then only one entry for C appears (the highest-scoring interaction), and other interactions are not returned as duplicate entries for C.
3. Given results are ranked, when the first result is inspected, then it has the highest similarity of all contacts returned.

---

### L2-016: Why-this-match snippet
**Traces to:** L1-004

Each returned contact shall include a "why this match" snippet (the `matchedText`, ≤ 240 chars, trimmed on token boundaries) rendered on the result card.

**Acceptance Criteria:**
1. Given a match sourced from an interaction with long text, when the result is rendered, then `matchedText` is at most 240 characters and ends at a word boundary followed by `…` if truncated.
2. Given a featured (top) result, when screen `2. Search Results` renders it, then the "WHY THIS MATCH" block (id `fcWhy` / `GMFxy` in the design) is shown above the interaction pills.
3. Given any result, when rendered, then the snippet is visually distinguishable from the contact's role text by color `$foreground-secondary` versus `$foreground-primary`.

---

### L2-017: Similarity score tier visualization
**Traces to:** L1-004

The numeric similarity shall be shown next to each result and color-coded into tiers: High (≥ 0.90) using `Score High` (`0QZrm`), Mid (0.70–0.899) using `Score Mid` (`lzlpQ`), Low (< 0.70) using `Score Low` (`fiEpD`).

**Acceptance Criteria:**
1. Given similarity 0.96, when rendered, then the `Score High` component is used and the numeric label reads `0.96`.
2. Given similarity 0.87, when rendered, then the `Score Mid` component is used.
3. Given similarity 0.62, when rendered, then the `Score Low` component is used.

---

### L2-018: Results sortable by similarity or recency
**Traces to:** L1-004

The sort control on the results screen (`sortBtn` / `BQz5F`) shall toggle between `Similarity` (default) and `Most recent interaction`.

**Acceptance Criteria:**
1. Given the default state, when results are returned, then the sort label reads `Similarity` and results are ordered by similarity descending.
2. Given the user taps the sort control and chooses `Most recent`, when results re-render, then results are ordered by the `occurredAt` of the matched-interaction descending.
3. Given a re-sort, when the "contacts matched" count is displayed, then it remains unchanged.

---

### L2-019: Search pagination and virtualization
**Traces to:** L1-004

Search shall return up to 50 results per page. The client shall virtualize the list when the count exceeds 50.

**Acceptance Criteria:**
1. Given a query producing 120 matches, when the first page is requested, then 50 results are returned with `nextPage=2`.
2. Given the results list on the client exceeds 50 rows, when the user scrolls, then only the visible rows plus a small buffer are present in the DOM (virtualization).

---

### L2-020: Empty-query and zero-result states
**Traces to:** L1-004

**Acceptance Criteria:**
1. Given the user submits an empty query, when the backend receives it, then the response is `400 Bad Request` with a validation error.
2. Given zero results, when the results screen is shown, then a zero-state panel explains that no contacts matched, offers to open Ask mode, and the "contacts matched" count reads `0`.

## Conversational AI assistant ("Ask RecallQ")

### L2-021: Ask mode accepts natural-language questions
**Traces to:** L1-005

The Ask screen shall provide an input bar (matching `inputBar` / `tUHxK` in `ui-design.pen`) with a text field, plus button, microphone button, and a gradient send button. Submitting a question shall POST to `/api/ask` and stream the assistant's answer.

**Acceptance Criteria:**
1. Given the Ask screen is open, when the user types `"who should I talk to about a Series B?"` and taps Send, then the message appears as a right-aligned purple bubble (matching `uqBubble` / `MeWfk`) and the server receives the request.
2. Given the send button, when the input is empty, then the send button is disabled.
3. Given the input bar, when the viewport is XS, then it is pinned to the bottom of the screen with the blurred fade overlay (`inputArea` / `KitkD`).

---

### L2-022: Assistant answers stream to the client
**Traces to:** L1-005

The assistant answer shall be streamed via server-sent events (SSE) or HTTP chunked streaming. The UI shall progressively render tokens into the answer bubble (matching `ansBubble` / `6rsdj`).

**Acceptance Criteria:**
1. Given a question is submitted, when the server starts generating, then the first token is delivered to the client within 1.5s (p95).
2. Given streaming has started, when tokens arrive, then the answer bubble renders them progressively without waiting for completion.
3. Given the stream completes, when the final token is received, then the request is closed and the cited-contact mini-cards become interactive.

---

### L2-023: Answer cites contacts as mini-cards
**Traces to:** L1-005

The assistant shall include a list of contact citations grounding its answer. Each citation shall render as a mini-card (matching `mini1`/`mini2`/`mini3` on screen `4. AI Ask Mode`) with avatar, name, short body, and similarity score.

**Acceptance Criteria:**
1. Given the answer references 3 contacts, when the response completes, then 3 mini-cards render inside the answer bubble, each with the contact's avatar gradient, name, role, and similarity score from the retrieval step.
2. Given a mini-card is tapped, when routed, then the contact detail screen for that contact opens.
3. Given a mini-card's score is ≥ 0.90, when rendered, then the `Score High` component (`0QZrm`) is used.

---

### L2-024: Follow-up suggestion chips
**Traces to:** L1-005

After an answer, the system shall display up to 3 follow-up suggestion chips (matching `fu1`/`fu2`/`fu3` on screen `4. AI Ask Mode`).

**Acceptance Criteria:**
1. Given an answer has been produced, when the UI settles, then up to 3 follow-up chips appear under a `FOLLOW-UP` label.
2. Given the user taps a chip, when routed, then its text is posted as the next user question.
3. Given fewer than 3 follow-ups are available, when the UI renders, then only those available are shown (no placeholder chips).

---

### L2-025: Conversation history within a session
**Traces to:** L1-005

The Ask screen shall retain the current user+assistant exchange history until the user navigates away or starts a new session via the `+` / `newBtn` action.

**Acceptance Criteria:**
1. Given the user has had 3 exchanges in Ask mode, when they scroll up, then all prior exchanges are visible.
2. Given the user taps the `+` new-session button (`newBtn` / `dEvpw`), when confirmed, then the screen clears to the greeting state.
3. Given the user navigates away to a contact detail and returns via the bottom nav, when they return within the same app session, then the prior exchange history is retained.

## Smart Stacks

### L2-026: Home surfaces at least three Smart Stacks
**Traces to:** L1-006

The home screen shall display a horizontally scrollable "Smart stacks" row (matching `stacksRow` / `e8zNf`) with at least three stacks: count + label (e.g., `27 AI founders`, `14 Intros owed`, `9 Close friends`).

**Acceptance Criteria:**
1. Given a user with indexed data, when the home screen renders, then at least 3 stack cards are shown with a count and a label.
2. Given the count is 0 for a stack, when rendered, then that stack is hidden (not shown as `0 …`).
3. Given the viewport is XS, when the row renders, then it is horizontally scrollable and the first card starts at `x=24` matching the mobile design padding.

---

### L2-027: Tapping a stack opens a filtered list
**Traces to:** L1-006

**Acceptance Criteria:**
1. Given the user taps the `AI founders` stack, when the view transitions, then the search results screen opens with a pre-applied filter and the query chip reads `AI founders`.
2. Given a filtered list is shown, when the user pulls back, then they return to the home screen with scroll position preserved.

---

### L2-028: Stack definitions are user-scoped and recomputed
**Traces to:** L1-006

Each stack shall be defined by either a tag filter, a saved semantic query, or an AI classification. Stack counts shall be recomputed when underlying contacts or interactions change.

**Acceptance Criteria:**
1. Given a stack defined as the saved semantic query `"AI founders"`, when a new contact matching that query is added, then the count reflected on the next home-screen render increases by 1.
2. Given a user deletes a contact counted in the `Intros owed` stack, when home renders next, then the count decreases accordingly.
3. Given user A's stacks, when user B logs in, then user B sees only their own stacks (data isolation).

## Proactive AI suggestions

### L2-029: Home displays an AI suggestion card when signal exists
**Traces to:** L1-007

The home screen shall include an AI suggestion card (matching `ai suggestion` / `Cm94Y`) with a heading `AI SUGGESTION`, a body sentence (e.g., `You met 3 AI founders last week — shall I find similar investors?`), a primary pill action, and a `Dismiss` ghost action.

**Acceptance Criteria:**
1. Given the user has had at least one qualifying interaction pattern in the last 30 days, when the home screen renders, then an AI suggestion card is displayed with a non-empty body.
2. Given no qualifying signal exists, when home renders, then the card is hidden and the `Smart stacks` section shifts up to occupy the space.
3. Given the card's primary action is tapped, when routed, then the appropriate search or Ask view opens seeded with the suggestion's implied query.

---

### L2-030: Dismissing a suggestion suppresses it
**Traces to:** L1-007

**Acceptance Criteria:**
1. Given a suggestion shown, when the user taps `Dismiss`, then the suggestion is suppressed for at least 7 days and a `dismissedAt` timestamp is persisted per user per suggestion-key.
2. Given a dismissed suggestion, when the 7 days pass and the signal still exists, then the suggestion may re-appear.
3. Given two different qualifying signals, when one is dismissed, then the other shall still be eligible to display the next day.

## AI-generated relationship summaries

### L2-031: Contact detail shows a relationship summary
**Traces to:** L1-008

The contact detail screen shall render a `RELATIONSHIP SUMMARY` card (matching `aiSummary` / `GwZQR`) with an AI-generated paragraph plus a stat band containing interaction count, sentiment tier, and days-since-last-interaction.

**Acceptance Criteria:**
1. Given a contact with ≥ 1 interaction, when the detail screen renders, then a summary paragraph is shown, `Interactions` is the contact's lifetime interaction count, `Sentiment` is one of `Warm`/`Neutral`/`Cool`, and `Since last` is the human-readable time since the most recent interaction (e.g., `3 days`).
2. Given a contact with 0 interactions, when the detail screen renders, then the summary card shows a "not enough data yet" state and the stat band shows `0`, `—`, `—`.
3. Given the user taps `Refresh` (text `I0Qo1` in the design), when the request completes, then the summary is regenerated and the paragraph updates.

---

### L2-032: Summary refresh is rate-limited and cached
**Traces to:** L1-008

Refresh shall be rate-limited to at most once per 60 seconds per contact per user.

**Acceptance Criteria:**
1. Given a user taps `Refresh` twice within 60 seconds, when the second request is made, then the response is `429 Too Many Requests`.
2. Given a summary has been generated, when the detail screen is opened again within 1 hour with no new interaction, then the cached summary is returned without invoking the LLM.

---

### L2-033: Summary auto-refresh on new interaction
**Traces to:** L1-008

When an interaction is added, updated, or deleted for a contact, the cached summary for that contact shall be invalidated and regenerated in the background.

**Acceptance Criteria:**
1. Given a cached summary and a new interaction is added, when the background job runs, then the summary is regenerated and the stat band reflects the new interaction count and "since last" time.
2. Given regeneration fails, when the user opens the detail screen, then the last valid summary is shown with a small `stale` indicator and automatic retry occurs.

## Mobile-first responsive UI

### L2-041: XS layout mirrors the mobile design
**Traces to:** L1-011

At XS (<576px), every screen shall match the structure, spacing, typography, and component layout of the corresponding frame in `ui-design.pen` within a 4px tolerance for spacing and a 1px tolerance for strokes.

**Acceptance Criteria:**
1. Given viewport width 390px, when the home screen renders, then the top bar padding, hero block position, search bar, suggestion chips row, AI suggestion card, Smart stacks row, and bottom nav appear at the same vertical positions as frame `1. ## Security, privacy, and data protection

### L2-052: Password hashing and storage
**Traces to:** L1-013

Passwords shall be hashed with Argon2id (preferred) or bcrypt cost ≥ 12. Hashes shall never appear in logs, error messages, or API responses.

**Acceptance Criteria:**
1. Given a new registration, when the account is stored, then the database row contains only a hash and a salt — no plaintext password.
2. Given a search of logs, when inspected, then no log line contains the string `password` followed by a value.

---

### L2-053: Server-side input validation
**Traces to:** L1-013

All inputs on all endpoints shall be validated server-side (length, type, enum membership, format). The client shall also validate, but the server shall not rely on it.

**Acceptance Criteria:**
1. Given a request bypassing the client (e.g., curl) with an oversize `displayName`, when the server processes it, then the response is `400 Bad Request` with a validation error.
2. Given a request with a `type` outside `{email, call, meeting, note}`, when POSTed, then the response is `400 Bad Request`.
3. Given a request with HTML/script payload in `content`, when persisted, then the content is stored as raw text and, when rendered on the client, is escaped so that no script executes (XSS prevention).

---

### L2-054: CSRF and auth transport
**Traces to:** L1-013

If cookies are used for auth, CSRF protection shall be enforced via the double-submit-cookie or SameSite=Strict pattern. If bearer tokens are used, tokens shall be sent via `Authorization: Bearer` and never as query parameters.

**Acceptance Criteria:**
1. Given cookie auth is in use, when a POST/PATCH/DELETE is made without a matching CSRF token, then the response is `403 Forbidden`.
2. Given bearer auth is in use, when a token appears in a URL query string, then the server rejects the request with `400 Bad Request` and logs a security event.

---

### L2-055: Rate limiting and abuse protection
**Traces to:** L1-013

Login, registration, search, ask, and summary-refresh endpoints shall enforce per-user and per-IP rate limits.

**Acceptance Criteria:**
1. Given the search endpoint limit of 60 requests/minute per user, when the 61st request is made in the same minute, then the response is `429 Too Many Requests` with a `Retry-After` header.
2. Given the Ask endpoint limit of 20 requests/minute per user, when exceeded, then the response is `429`.

---

### L2-056: Per-user data isolation
**Traces to:** L1-013

Every query issued against contacts, interactions, embeddings, summaries, and suggestions shall include an owner-scope predicate. The owner scope shall be enforced in the data layer (EF Core global query filter) so a missed filter at the endpoint layer does not leak data.

**Acceptance Criteria:**
1. Given the EF Core model, when the application starts, then global query filters are registered on `Contact`, `Interaction`, `ContactEmbedding`, `InteractionEmbedding`, `RelationshipSummary`, and `Stack` that filter to `CurrentUserId`.
2. Given a hand-crafted SQL query bypasses ORM filters, when reviewed, then every such query includes an explicit `owner_user_id = @userId` predicate.
3. Given an integration test with two users, when user B asks for all contacts, then user A's rows never appear in the response.

---

### L2-057: Encryption in transit and at rest
**Traces to:** L1-013

All HTTP traffic shall require TLS 1.2+. Database-level encryption at rest shall be enabled in production deployments.

**Acceptance Criteria:**
1. Given a production deployment, when the server starts, then only TLS 1.2+ is accepted and HTTP requests are redirected to HTTPS with HSTS headers set.
2. Given the managed database, when provisioned in production, then encryption at rest is enabled.

**Acceptance Criteria:**

1. Given the user submits valid credentials with `rememberMe=true`, when the response is issued, then the auth cookie is set with `Max-Age` ≥ 30 days (or, for bearer tokens, a refresh token valid for 30 days is issued) and `Secure; HttpOnly; SameSite=Lax|Strict` attributes are present.
2. Given the user submits valid credentials with `rememberMe=false`, when the response is issued, then the auth cookie is a session cookie (no `Max-Age`/`Expires`) or the bearer token expires within 24 hours and no refresh token is issued.
3. Given a `rememberMe=true` session and the user closes and reopens the browser within 30 days without other invalidation, when they revisit the app, then they are still authenticated.
4. Given the user explicitly logs out (L2-004), when prior `rememberMe` cookies/tokens are reused, then the response is `401 Unauthorized`.
5. Given the login form on XS viewport, when rendered, then the `Remember me` checkbox is left-aligned and the `Forgot password?` link is right-aligned on the same row, matching the design.
6. Given the checkbox is keyboard-focused and `Space` is pressed, when toggled, then the visual filled state changes and the `aria-checked` attribute reflects the state.

### L2-086: "Forgot password?" link on login
**Traces to:** L1-001

The login screen shall include a `Forgot password?` link (matching `forgot` / `KTo4x` in the design) that routes the user to the Forgot Password screen.

**Acceptance Criteria:**
1. Given the user is on the login screen, when they activate the `Forgot password?` link (tap, click, or `Enter` while focused), then the app navigates to the Forgot Password screen (`7. Forgot Password` in `ui-design.pen`).
2. Given the link, when inspected, then it has an accessible name `Forgot password?` and is rendered with color `$accent-tertiary` and `font-weight: 600`.
3. Given the user has typed an email in the login email field, when they navigate to Forgot Password, then the same email value is pre-filled in the Forgot Password email field.

### L2-087: Forgot Password request page
**Traces to:** L1-001

The system shall provide a Forgot Password page (matching `7. Forgot Password` in `ui-design.pen`) that accepts an email address and submits a password-reset request. The page shall be mobile-first and shall render correctly at all five viewport breakpoints.

**Acceptance Criteria:**
1. Given the user submits a valid email to `POST /api/auth/forgot-password`, when the request is processed, then the response is always `202 Accepted` regardless of whether the email matches an account (no account-existence leak), and a success state is rendered telling the user to check their inbox.
2. Given the email matches a registered account, when the request is processed, then a single-use, time-limited reset token (valid ≤ 60 minutes, ≥ 256 bits of entropy, stored as a hash on the server) is generated and an email is dispatched containing a reset link `{appUrl}/reset-password?token={token}`.
3. Given the email does not match any account, when the request is processed, then no email is sent but the response and rendered UI are identical to the matching case.
4. Given the user submits an empty or syntactically invalid email, when validated, then the response is `400 Bad Request` with a validation error and the page shows an inline error.
5. Given the user taps `← Back to sign in`, when routed, then the login screen is shown.
6. Given viewport widths 390px / 576px / 768px / 992px / 1200px, when the page renders, then the layout matches `ui-design.pen` at XS within 4px tolerance and at SM/MD/LG/XL the form is centered with a max content width of 420px.
7. Given the endpoint, when 5 forgot-password requests are made for the same email or from the same IP within 60 seconds, then the 6th is rejected with `429 Too Many Requests` and a `Retry-After` header.

### L2-088: Password reset email content and security
**Traces to:** L1-001

The reset email shall be sent over an authenticated transport (e.g., a transactional email provider with DKIM/SPF), shall contain only a single-use reset link, and shall not include the user's password or any contact data.

**Acceptance Criteria:**
1. Given an email is dispatched, when its body is inspected, then it contains the reset link, the link's expiry duration, and a "if you didn't request this, ignore this email" note — and contains no password, password hash, or contact data.
2. Given the reset link is used once successfully, when it is reused, then the response is `400 Bad Request` with an "expired or invalid token" message.
3. Given the reset token has expired (> 60 minutes since issue), when used, then the response is `400 Bad Request` with the same generic "expired or invalid token" message (no token-existence leak).
4. Given a reset is successfully completed, when prior outstanding reset tokens for the same user exist, then they are invalidated server-side.
5. Given an attacker submits arbitrary or guessed tokens, when the endpoint is called, then it is rate-limited to ≤ 10 attempts per IP per minute and returns `429` thereafter.

### L2-089: Reset Password page (consume token)
**Traces to:** L1-001

The system shall provide a Reset Password page reached via the email link `?token=…`. The page shall capture a new password (with confirm field), validate the token server-side, set the new password, invalidate all existing sessions for the user, and redirect to login.

**Acceptance Criteria:**
1. Given a valid, unexpired token and a new password meeting the password policy from L2-001, when the user submits to `POST /api/auth/reset-password` with `{ token, newPassword }`, then the response is `200 OK`, the password hash is updated, and all existing sessions/refresh tokens for that user are invalidated.
2. Given the new password fails the policy (length < 12, missing letters/digits, etc.), when submitted, then the response is `400 Bad Request` with a validation error and the password is not changed.
3. Given the new password and the confirm field do not match on the client, when the user attempts to submit, then the submit button is disabled and an inline error is shown — no request is sent.
4. Given a missing, malformed, or expired token in the URL, when the page loads, then a clear error state is shown with a `Request a new link` button that routes back to the Forgot Password page.
5. Given a successful reset, when the redirect occurs, then the user lands on the login screen with a non-PII confirmation toast (e.g., `Password updated. Please sign in.`).
6. Given the page on XS, when rendered, then it matches the visual style of `7. Forgot Password` (same surface tokens, ambient glows, `Button Primary` for submit) and the password fields use the same input styling as `6. Login`.
7. Given the password input, when rendered, then a show/hide toggle is present, the field has `autocomplete="new-password"`, and `aria-label` describes the field.
